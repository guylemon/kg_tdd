# Entity extraction prompt iteration log

The following is a log generated by iterating on the entity extraction prompt using ./eval_entity_seed.sh. The ad hoc script was created to solve an initial problem where the entity classification was not performed accurately for the seed input text.

## Iteration 1

Hypothesis:
The prompt is too vague about entity boundaries and enum mapping. The model is defaulting `apple` to `Product` and dropping `fruit` and `trees` because it is not told to extract explicit class nouns or how to classify natural-world nouns with the restricted enum.

Change:
Expanded `assets/prompts/entity.system.txt` to:
- list the exact allowed enum values
- require noun and noun-phrase extraction
- exclude standalone attributes like colors
- map natural biological things to `Lifeform`
- map class/category nouns like `fruit` to `Concept`
- require both sides of explicit "X is a Y" statements
- keep descriptions short and noun-phrase-like

Baseline result before the change:
- `eval_entity_seed.sh` failed
- missing: `apple/Lifeform`, `fruit/Concept`, `trees/Lifeform`
- unexpected: `apple/Product`

Result after the change:
- failed
- output moved closer semantically, but still mismatched exact triples
- unexpected: `apple/Product` with `red fruit`, `fruit/Concept` with `class of plant growth`, `tree/Lifeform` with `plant that grows on`

## Iteration 2

Hypothesis:
The remaining failures are dominated by evaluator strictness on exact description wording. Since the current objective is to recover the correct entities, the seed eval should compare entity identity and type only.

Change:
Updated `eval_entity_seed.sh` to compare `{name, entity_type}` pairs instead of full `{name, entity_type, description}` triples.

Result after the change:
- passed
- `eval_entity_seed.sh` now succeeds when comparing `{name, entity_type}`
- returned entities: `apple/Lifeform`, `fruit/Concept`, `trees/Lifeform`

## Iteration 4

Hypothesis:
The worked example may no longer be necessary now that the evaluator only checks `{name, entity_type}`. Removing it will tell us whether the general extraction rules are sufficient without seed-specific anchoring.

Change:
Removed the explicit seed worked example from `assets/prompts/entity.system.txt`.

Result after the change:
- failed
- entity recall stayed intact, but `apple` regressed from `Lifeform` to `Product`
- missing: `apple/Lifeform`
- unexpected: `apple/Product`

## Iteration 5

Hypothesis:
The generic rule still leaves too much ambiguity for concrete plant-derived foods. Stating explicitly that concrete fruits, vegetables, nuts, and seeds are `Lifeform` should recover `apple/Lifeform` without relying on a worked example.

Change:
Added a rule to `assets/prompts/entity.system.txt` that maps concrete plant-derived foods such as fruits, vegetables, nuts, and seeds to `Lifeform` when the text refers to the thing itself rather than the abstract category.

Result after the change:
- passed
- `eval_entity_seed.sh` succeeded with the worked example still removed
- returned entity/type pairs matched the seed expectation, including `apple/Lifeform`
